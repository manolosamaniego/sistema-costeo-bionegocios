# Flujo de soporte

## Objetivo

Atender fallas, dudas o mejoras de clientes sin improvisación y sin depender de que el equipo original esté físicamente disponible.

## Qué debe enviar el cliente

- captura del error o comportamiento
- archivo JSON del costeo
- versión de la app
- archivo de marca si aplica
- explicación breve de lo que estaba haciendo

## Flujo de atención

1. recibir la incidencia
2. registrar la incidencia
3. reproducir el caso en la matriz
4. identificar el módulo afectado
5. corregir el código
6. probar con el mismo JSON
7. documentar el ajuste
8. generar nueva versión
9. entregar instalador o instrucciones
10. cerrar la incidencia

## Tipos de soporte

### Soporte correctivo

Cuando algo ya no funciona o produce un resultado inconsistente.

### Soporte funcional

Cuando el cliente no entiende un módulo o requiere ayuda de uso.

### Soporte evolutivo

Cuando se requiere un cálculo nuevo, una mejora de interfaz o un nuevo módulo.

## Tiempo recomendado de registro

Cada incidencia debe registrar:

- fecha
- cliente
- módulo
- síntoma
- archivo recibido
- versión afectada
- versión corregida
- estado

## Regla clave

Nunca corregir directamente sobre una copia enviada al cliente.
Siempre corregir primero en la matriz.
